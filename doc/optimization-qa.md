# 持续优化自问自答

这份文档不是当前 sprint 的待办单，而是站在“产品负责人 + 架构师 + 演示答辩视角”对项目做的一轮自问自答。

目标只有一个：让后续任何一位 coder 接手时，都能快速判断“下一步最值得优化什么，为什么”。

最后更新：2026-03-17

## 1. 如果只允许继续做 3 件事，最该做什么？

### Q1：为什么不是先继续堆设备模板？

因为模板数量已经足够支撑“混合设备监测系统”的展示逻辑了，当前瓶颈不在“有没有更多设备”，而在：

- 真实模型模式还没有真正稳定可用
- Android 客户端还没有完成真机联调
- PC 客户端虽然更像产品了，但还缺更细的版本提示和日志入口

结论：

- 先补“系统可信度和可演示性”
- 再补“设备广度”

### Q2：最值得优先推进的 3 件事是什么？

1. 真实模型模式联通验收  
   原因：这是最直接影响“AI Agent 是否像真 AI”的部分。
2. Android 真机联调  
   原因：这决定“手机端客户端”能否从“可构建”升级成“可落地”。
3. PC 客户端最后一轮产品化补强  
   原因：这会直接提升演示体验，也更符合 C 端软件形态。

## 2. PC 和移动端客户端，下一步最值得优化什么？

### Q3：个人 PC 客户端已经有 GUI 了，下一步还缺什么？

这一轮已经补上了：

- 托盘运行 / 最小化到系统托盘
- 开机自启动
- 网络异常时自动重试与目标网关预览

接下来更值得补的是：

- 客户端版本号与更新提示
- 更细的本地运行日志入口
- 更清楚的“当前 release 是脚本版还是 EXE 版”

建议优先顺序：

1. 客户端版本号与更新提示
2. 更细的本地运行日志入口
3. release 形态说明
4. 更细的本地错误导出入口

### Q4：移动端已经有 APK 工程了，最该补什么？

最该补的是“真机联调闭环”，不是继续加页面样式。

优先事项：

- 真机安装验证
- 手机到 PC 网关的局域网联通验证
- 后台运行策略验证
- 断网 / 锁屏 / 切后台情况下的上报行为验证

如果这些不做，APK 只能算“能构建”，还不能算“能落地”。

### Q5：移动端要不要继续做得更复杂，比如登录、历史页、告警页？

短期不建议。

因为当前手机端在系统中的职责是：

- 采集端
- 上报端
- 简易本地状态可视化

不是完整监控平台。

如果把大量 dashboard 功能搬到手机端，会和主页面职责重复，还会显著提高开发量。

## 3. AI Agent 这条线，后面应该怎么进化？

### Q6：当前聊天已经能问答了，为什么还要继续收敛 Tool / Skill？

因为现在的系统虽然“能回答”，但还没有把“能力边界”和“调用路径”彻底统一。

后续如果要接：

- OpenAI function calling
- MPC Skill 平台
- 真实百炼 / 阿里云技能编排

就需要让聊天主流程尽量稳定地走统一 adapter，而不是一部分逻辑在规则问答里，一部分在 Tool 层，一部分又旁路调用。

### Q7：真实模型接入后，最容易翻车的点是什么？

最容易翻车的不是 API 连不上，而是：

- 提示词不稳定，回答太散
- 工具调用过多，成本高
- 工具调用过少，回答空泛
- 设备上下文不够明确，导致答非所问

所以真实模型联调不应该只看“有没有回复”，而应该重点看：

- 是否引用了当前选中设备
- 是否能区分总览问题和单设备问题
- 是否在异常设备场景下给出结构化理由
- 是否能稳定回退到本地规则

### Q8：要不要让本地 Ollama 和真实模型完全共用一套提示词？

可以共用主框架，但不要强行完全一致。

原因：

- 本地 7B 模型更需要短 prompt、强约束
- 真实模型更适合 richer context 和 tool schema

更好的方式是：

- 共享一套任务分层
- 按模型类型分不同提示词模板

## 4. 网关与后端，哪一块最值得继续做？

### Q9：共享 HTTP 网关现在已经能用了，为什么还要继续补 backend manager？

因为现在只是“能跑”，还不算“稳”。

这一轮已经补上了：

- PID 文件与 PID 存活识别
- 健康检查接口
- 启动失败后的可见错误
- 健康失败后的自动重启

接下来更值得补的是：

- 更清楚的日志和故障排查入口
- 端口占用和服务启动失败时的更友好提示
- 如果继续工程化，再考虑服务化安装而不是脚本级 wrapper

这些东西不会直接提升页面观感，但会显著降低演示时翻车概率。

### Q10：MQTT 现在值不值得接？

值得评估，但不建议抢在前 3 优先级之前。

适合接 MQTT 的原因：

- 更符合多设备持续上报场景
- 更适合弱网和断线重连
- 更像真实 IoT 项目

不适合立刻接的原因：

- 当前 HTTP push 已经够支撑单机和局域网演示
- MQTT 会引入 broker、订阅、QoS、部署说明等额外复杂度

结论：

- 把 MQTT 当“第二通道”
- 不要把它当成当前必须重写的主链路

## 5. 数据层和可观测性，什么时候该升级？

### Q11：现在还用 JSON 文件，会不会太原始？

会，但在当前阶段仍然合理。

因为这个项目当前更偏：

- 演示系统
- 课程设计 / 答辩项目
- 单机或轻量多端接入

JSON 的优势是：

- 简单
- 可读
- 好调试

但如果接下来要强调“工程化”，下一步最自然的是 SQLite，而不是直接上重数据库。

### Q12：什么时候该从 JSON 升级到 SQLite？

当你遇到下面 3 类需求时，就该升：

- 需要按设备 / 时间范围快速查询历史
- 需要长期保存而不是只看最近数据
- 需要为 GitHub 状态页、报告导出、异常统计提供稳定数据源

## 6. 面试 / 答辩视角，还能怎么加分？

### Q13：如果面试官只看 5 分钟，什么最能体现这个项目的成熟度？

最加分的不是“功能再多一个”，而是下面几件事：

- GitHub 首页一眼能看懂项目是什么
- Projects 看板能看出你真的持续推进过
- 文档能看出你有架构意识，不只是堆代码
- 客户端是可交付的，不只是命令行 demo
- AI、设备、网关、展示层边界清楚

### Q14：答辩版还应该补什么材料？

建议补 4 类材料：

- 页面截图
- 架构图
- 真实设备接入流程图
- 一份简短讲解词

这几样对“展示效果”的提升通常大于再多写一个功能点。

## 7. 如果继续做一周，最合理的节奏是什么？

### Q15：推荐的一周推进顺序是什么？

1. 真实模型模式联通与稳定性验收
2. Android 真机联调与客户端小修
3. PC 客户端版本提示 / 日志入口
4. 移动端连接状态与本地说明增强
5. GitHub 展示页、答辩素材和截图整理

这个顺序的核心思想是：

- 先把系统可信度补稳
- 再补体验
- 最后补展示包装

## 8. 一句话结论

这个项目现在最值得继续强化的，不是“再做更多功能”，而是把已经有的能力变得：

- 更稳
- 更像真实产品
- 更容易展示
- 更容易交接
